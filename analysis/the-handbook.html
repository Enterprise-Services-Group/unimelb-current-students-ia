The Handbook — Deep-Dive

The biggest single dependency in the University's web estate, and the one governed to the lowest web standard. 7,819 contextual inbound links from across 15+ domains — more than any other system — but SPA-only with no server-rendered fallback, blank titles on every page, and no "enrol" action. June 2026.

Executive summary

The Handbook (handbook.unimelb.edu.au) is the system that defines course structure, subject prerequisites, points, and entry requirements for every degree the University offers. It is the #1 dependency in the estate:

  • **7,819 contextual inbound links** from 15+ domains — more than my.unimelb (187), ask.unimelb (396), or Careers Online (115) combined
  • **8,323 outbound links** — the Handbook is the estate's second-most prolific linker after study.unimelb
  • Referenced on **~830 hub enrolment pages** (195 direct links from the course-planning toolchain alone)
  • **2,134 pages** — 161 subjects, 105 courses, plus structure and search pages

But the Handbook is governed to a visibly lower web standard than the rest of the estate. It's built on a different platform, operates as a single-page application with no server-rendered fallback, has blank </code> tags on every subject and course page</strong>, carries <strong>non-descriptive anchor text</strong> at scale, and provides <strong>no "Enrol in this subject" action</strong> — the single most important task a student would want to perform from a subject page. For ~1,470 of 2,134 crawled captures, the page delivered to the crawler is an 887-byte Imperva bot-block shell — meaning search engines, social-media link previews, and non-JS users see nothing at all.</p> <p>The fix is not a rebuild. It's SSR, metadata templates, browsable indexes, and one enrol action per subject page. But those links are generic (bare URLs, "current students" labels). None say "Enrol in this subject."</p> <hr> <h2>The three failures</h2> <h3>1. SPA-only with no server-rendered fallback `[HIGH]`</h3> <p>The Handbook is a single-page application. When a search engine, social-media crawler, or non-JS user-agent requests a subject or course page, the server returns JavaScript that must be executed to render content. The Imperva WAF adds a second layer: ~1,470 of 2,134 crawled captures are 887-byte bot-block shells — the WAF challenged the crawler and the crawler couldn't execute JS to pass the challenge.</p> <p>The result:</p> <ul> <li>**Search engines see blank pages with no content.** Every handbook page is effectively invisible to Google.</li> <li>**Social media link previews show nothing.** Share a handbook subject link on Slack/Teams/LinkedIn — no title, no description, no image.</li> <li>**Screen readers and accessibility tools fail.** No server-rendered content = no accessible DOM.</li> <li>**266 pages are "orphans"** (161 subjects + 105 courses) — but this is an artifact of the bot-block. The pages exist; the crawler couldn't see who links to them.</li> </ul> <h3>2. Blank `<title>` on every subject and course page `[HIGH]`</h3> <p>Every subject page and course page in the Handbook has an empty or generic <code><title></code> tag. This is the single most impactful accessibility + SEO + findability failure in the estate by volume. When a student bookmarks a subject, the bookmark reads nothing. When they search their browser history, nothing appears. When Google indexes the page (if it can), the search result has no title.</p> <h3>3. No "Enrol in this subject" action `[HIGH]`</h3> <p>A student looking at a subject page in the Handbook wants to do one thing: enrol in it. The Handbook describes the subject — points, prerequisites, assessment, timetable — but provides <strong>zero path to enrol</strong>. The "describe but don't transact" pattern that defines the estate's worst faults is most acute here: the page a student lands on to decide <em>whether</em> to enrol has no way to <em>actually</em> enrol. §6.7 of the improvements register calls for exactly this action.</p> <hr> <h2>Anchor text: non-descriptive at scale</h2> <p>The Handbook carries thousands of links, but anchor text is generic — "current students," "handbook," bare URLs. This is the #2 findability finding in §2.6. Every link from the Handbook to the student hub should say what it does: "Enrol in subjects," "Check census dates," "Manage your timetable." Instead they're chrome-level labels.</p> <hr> <h2>The orphan problem — mostly an artifact</h2> <p>266 handbook pages appear as orphans (no inbound links). 161 are subjects, 105 are courses. But this is substantially a <strong>crawl artifact</strong>: the WAF blocked the crawler from seeing the full link graph. The pages are linked — just not visible through the bot-block. However, the artifact itself is the problem: if the crawler can't see inbound links, neither can Google.</p> <hr> <h2>Recommendations</h2> <h3>1. Add server-rendered fallback for every subject and course page `[HIGH · large]`</h3> <p>Implement SSR so every /subjects/<em> and /courses/</em> page delivers a complete HTML document with title, h1, meta description, prerequisite list, points, and assessment structure — before any JavaScript executes. This fixes search engine visibility, link previews, accessibility, and the bot-block problem in one change.</p> <h3>2. Give every subject and course page a descriptive `<title>` `[HIGH · medium]`</h3> <p>Template: "MULT10001 Introduction to Life Sciences — Handbook | University of Melbourne". One template change touches all 2,134 pages. The highest-impact accessibility + SEO fix in the estate.</p> <h3>3. Add one "Enrol in this subject → my.unimelb" action to every subject page `[HIGH · medium]`</h3> <p>The action students want, on the page they use to decide. One CTA, one destination (my.unimelb enrolment), one template change.</p> <h3>4. Build browsable indexes — stop being search-only `[HIGH · medium]`</h3> <p>Subjects and courses are currently discoverable only via the Handbook's internal search. Add browsable A-Z indexes, by-faculty listings, and by-level groupings. These are the pages that search engines and students can navigate.</p> <h3>5. Fix non-descriptive anchor text `[MEDIUM · medium]`</h3> <p>Replace generic link labels ("current students," "handbook") with action-oriented labels ("Enrol in subjects," "Check prerequisites," "View course structure").</p> <h3>6. Confirm the bot-block serves a real HTML fallback to legitimate non-JS user agents `[MEDIUM · medium]`</h3> <p>The Imperva challenge should not block screen readers, accessibility tools, or link-preview bots. Configure WAF to serve SSR'd content to known-good non-JS user agents.</p> <hr> <h2>Linked improvements from the register</h2> <table> <tr><th>§</th><th>Improvement</th><th>Severity · Effort</th></tr> <tr><td>2.2</td><td>Surface live 2026 handbook subjects/courses via browsable indexes + give them a `<title>`</td><td>HIGH · medium</td></tr> <tr><td>5.3</td><td>Bring the Handbook up to the estate's web-standards baseline</td><td>HIGH · medium</td></tr> <tr><td>6.6</td><td>Treat handbook subject/course pages as JS-only — provide a server-rendered fallback</td><td>MEDIUM · large</td></tr> <tr><td>6.7</td><td>Wire an "Enrol in this subject" action onto every handbook subject & course page</td><td>HIGH · medium</td></tr> <tr><td>2.6</td><td>Fix non-descriptive anchor text on the Handbook</td><td>MEDIUM · quick-win</td></tr> <tr><td>5.7</td><td>Annotate the handbook crawl artifact so a false "14% orphaned" doesn't reach the deck</td><td>LOW · quick-win</td></tr> </table> <hr> <p><em>Built from: cross-site-flow.csv (7,819 inbound / 8,323 outbound links), orphans.csv (266 handbook orphans), page-weight.csv (0KB median = bot-block artifact), improvements-register.md (§2.2, §5.3, §6.6, §6.7, §2.6), and crawl/handbook/ (2,134 captures).
